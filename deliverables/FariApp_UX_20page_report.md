# FariApp 用户体验分析与交互补齐建议

生成时间：2026-07-01  
分析对象：`Builds/Android/FariApp.apk` 与当前 Unity 工程  
报告形式：20 页 Markdown 诊断稿

---

## 第 1 页：执行摘要

FariApp 当前已经具备一个相对完整的核心产品骨架：用户每天抽一张“今日神谕”牌，在对话中继续追问，也可以围绕朋友或自建关系档案发起双人关系占卜；个人中心中已经覆盖历史、反馈、通知、记忆、账号、会员等模块。从功能广度看，它不只是一个简单塔罗抽牌工具，而是在尝试做“陪伴式神谕 + AI 对话 + 关系记忆 + 社交邀请”的复合体验。

从用户体验看，产品最强的部分是沉浸气质。暗色背景、橙金强调色、占卜师角色图、卡牌翻开动作、每日仪式感，都很适合情绪陪伴与轻神秘学场景。设计参考图里，首页和翻牌流程很快能让用户理解“这是一个每天打开一次也值得的应用”。

目前最大的 UX 风险不是缺少功能，而是“功能已经很多，但用户第一次进入时不知道为什么要相信它、为什么要授权、为什么要继续”。应用声明了通讯录、通知、广告 ID、账单、外部存储等权限；注册流程要求生日、出生时间、出生城市；对话会产生 AI 记忆；好友功能涉及真实好友、虚拟好友、双人关系占卜、每日牌同步。每一项本身都有价值，但如果没有渐进解释，会让新用户感觉负担很重。

优先级最高的补齐方向有 8 个：

1. 统一品牌、语言和文案，去掉 `fariapp`、`Button`、测试歌词、葡语/中文/英文混杂等开发态痕迹。
2. 补一个极短的新手价值引导：每日牌、对话、朋友占卜、隐私控制各一句，不做营销页，但要降低理解成本。
3. 给生日、出生时间、出生城市、通讯录、通知、AI 记忆增加“为什么需要”的权限前置解释。
4. 对每日神谕、AI 对话、双人占卜增加等待态、失败态、重试态、离线态和本地保存提示。
5. 增强底部导航与跨页面返回规则，避免用户在多层弹窗/窗口栈中迷路。
6. 把对话入口做成产品主循环，而不是只依赖用户主动输入。
7. 把隐私与记忆管理前置为可信任资产，而不是藏在设置里。
8. 在发布前完成真机验收、包体优化、权限清理和商店素材一致性。

一句话结论：FariApp 的内容和功能已经像 MVP 后期，体验包装还停留在内部测试态。下一步不应继续堆功能，而应优先把“首访信任、核心闭环、状态反馈、隐私解释、文案一致性”打磨到可发布水平。

## 第 2 页：分析范围、证据与限制

本报告基于三类资料：

第一类是 Android APK 静态信息。通过 `aapt dump badging` 和 `aapt dump permissions` 读取到：包名为 `com.canchentechnology.fariapp`，版本 `0.1`，versionCode 为 `4`，targetSdkVersion 为 `35`，首包体积约 `162MB`。APK 内包含 Unity IL2CPP、Firebase Auth/Firestore/Messaging/Database、Unity IAP、Facebook SDK、YooAsset 热更新资源等。启动 Activity 为 `com.google.firebase.MessagingUnityPlayerActivity`，应用名当前显示为 `fariapp`。

第二类是 Unity 工程源码与文档。工程中可以看到 `TodayOracleUI`、`DialogUI`、`FriendUI`、`MyUI`、`UnlockProUI`、`MemoryManagementUI`、`NotionUI`、`FeedbackUI` 等完整模块；文档 `Fari界面视图功能提取.md` 记录了产品的信息架构和已实现功能；`剩余需要完成的事.md` 记录了发布、推送、IAP、真机验证等剩余事项。

第三类是 UI 视觉参考图。`Assets/GameData/Arts/UI/Example` 中的首页、翻牌、聊天、好友、历史、双人占卜等截图展示了目标视觉风格。设计气质统一，但部分截图仍保留测试文案、语言混杂和占位内容。

本次未完成的部分是“真机可用性走查”。当前没有连接 Android 设备或模拟器，无法实际安装 APK、点击路径、观察性能帧率、系统权限弹窗、键盘遮挡、弱网行为、推送点击落地页和 IAP 商店页。因此，本报告属于“静态包体 + 源码 + 视觉稿”的 UX 诊断，不等同于真机验收报告。

报告中的建议按用户体验风险排序，而不是按开发难度排序。P0 代表发布前必须处理；P1 代表会明显提升留存、信任或转化；P2 代表增强体验精致度和长期增长。

## 第 3 页：产品定位与核心体验判断

FariApp 的定位可以概括为：一个以每日塔罗神谕为入口，以 AI 神谕师对话为连续陪伴，以好友关系占卜和记忆系统为差异化能力的情绪洞察产品。

它不是传统工具型应用。用户不是为了“完成一个任务”而来，而是带着困惑、好奇、情绪波动或关系问题进入。因此它的 UX 关键不是效率最大化，而是“被理解、被引导、被安全地带着往前走”。这类产品的第一感受非常重要：如果开场像游戏，用户期待刺激；如果开场像心理咨询，用户期待严肃；如果开场像神秘学仪式，用户期待氛围和可信的解释。FariApp 目前最接近第三种。

核心体验闭环可以设计为：

1. 每天打开 App，看到今日主题。
2. 翻开一张牌，获得一句可理解的提醒。
3. 对这张牌继续追问，进入 AI 神谕师对话。
4. 对话中产生记忆、推荐追问或牌阵。
5. 用户把结果保存、回看、分享，或带入某个好友关系。
6. 第二天通过提醒、回访或明日线索再次回来。

当前工程中已经有这条闭环的许多实现：每日神谕、完整解读、深度对话、历史记录、TTS、记忆、通知、好友每日牌同步、双人关系占卜、Pro 额度门槛。问题在于每个模块之间的“用户理由”还没有完全串起来。用户能做很多事，但不一定知道下一步为什么值得做。

建议把产品的一句话价值聚焦成：每天抽一张牌，看见此刻的自己；有困惑时，让神谕师陪你把问题说清楚。好友和双人关系占卜是进阶价值，不应在首访时压到用户面前。

## 第 4 页：信息架构与导航

当前底部导航是四个主 Tab：今日神谕、对话、朋友、我的。这个结构合理，能覆盖“每日入口、深度互动、社交关系、个人资产”。但从设计图和 prefab 文案看，导航命名存在版本不一致：有的地方是 `Divination / Talk / Friend / Mine`，有的地方是“今日神谕 / 对话 / 一起玩 / 我的”，还有葡语截图中的 `Adivinhação / Interpretação / Social / Perfil`。这会直接影响用户对模块边界的理解。

建议统一为一套主语言。如果面向中文用户，推荐：

1. 今日
2. 对话
3. 朋友
4. 我的

如果面向英文用户，推荐：

1. Oracle
2. Chat
3. Friends
4. Me

“Divination”虽然准确，但对普通用户偏硬；“Talk”比“Chat”更口语，但在产品 UI 中 `Chat` 更通用；“Mine”是中式英语，英文界面应改为 `Me` 或 `Profile`。

导航层级也需要更明确。现在工程中大量页面通过 `UIModule.Instance.PopUpWindow<T>()` 弹出，体验上容易形成“窗口叠窗口”。例如朋友页进入搜索、创建好友、好友主页、编辑资料、关系占卜、通知设置后，用户需要稳定知道自己处于哪一层，返回会回到哪里。

建议增加 3 条导航规则：

1. 底部 Tab 永远只承载一级模块，Tab 切换时清理该模块内部临时状态，除非用户正在编辑表单。
2. 二级页面统一顶部栏：左侧返回、中间标题、右侧必要动作，不使用多个风格不同的返回入口。
3. 弹窗只用于确认、选择、权限说明和轻量菜单；完整页面不要伪装成弹窗。

P0：统一底部导航命名、图标选中态和禁用态。  
P1：补全页面栈规则和返回行为说明。  
P2：为复杂流程增加面包屑式进度条，例如创建好友、双人占卜。

## 第 5 页：首屏、启动与登录体验

启动流程从 `GameStart` 看，会先初始化 YooAsset、播放启动 Splash，再进入 App 场景；`AppStart` 会初始化通知和远程推送，然后弹出 `LoginUI`。这个技术流程是正常的，但用户体验上需要避免两个问题：等待不透明，以及登录过早。

启动页目前有沉浸式品牌素材，这是优点。但如果热更新、Firebase 初始化或资源加载较慢，用户需要知道“正在准备什么”。不要只展示静态神秘画面，建议加入轻量状态文案：

1. 正在唤醒神谕师
2. 正在整理今日牌面
3. 正在连接你的星图

这些文案要有超时兜底，超过 6 到 8 秒显示“网络较慢，可以稍后重试”或“继续进入离线模式”。已有热更新 PatchWindow，但体验上仍应把错误从开发日志转换成用户可理解的选择：重试、稍后、查看网络。

登录体验上，目前支持 Google、Apple、Facebook、邮箱、游客等路径，并会根据平台隐藏不可用入口。这是必要的。问题是：对一个占卜类应用，第一次打开就要求登录，会提高流失。用户还没体验价值，就要交出身份。

建议采用“先体验，后绑定”的登录策略：

1. 首次可直接进入今日神谕，使用匿名账号或本地临时身份。
2. 用户完成首次翻牌后，再提示“登录后可保存历史、同步记忆、找回记录”。
3. 只有涉及好友、云端同步、Pro、推送、跨设备历史时，才强制登录。

生日/出生信息同理。它对占星和个性化有价值，但不应成为首次体验阻塞。可以先让用户抽每日牌，再在完整解读或角色切换时提示“补充出生信息可获得更贴合的解读”。

P0：去掉登录页开发态文案和未配置按钮。  
P1：增加游客先体验路径。  
P1：把注册资料拆成渐进采集。  
P2：为启动等待和初始化失败做情绪化但明确的状态文案。

## 第 6 页：今日神谕与翻牌流程

今日神谕是 FariApp 的最佳首屏。它有强视觉、明确动作、每日频率和自然回访理由。设计图中的占卜师大图、橙金 CTA、底部导航都能快速建立产品气质。

目前可见问题主要是文案和交互反馈。示例中出现 `Puxe seu símbolo`、`Passado Presente Futuro`、`Flip open the card of destiny`、`Daily Tarot` 等不同语言。如果这不是刻意做多语言实验，发布前必须统一。占卜类产品非常依赖文字可信度，一旦用户看到语言混杂，会把它理解为粗糙或模板化。

翻牌交互建议补齐以下体验：

1. 卡牌可拖拽、轻扫或点击，不只依赖大按钮。
2. 选中某张牌时有轻微放大、光晕、震动反馈。
3. 翻开前给 0.5 秒“锁定选择”动画，让用户感觉这是自己的选择。
4. 翻开后先展示牌名和正逆位，再展示一句今日提醒，最后出现“查看完整解读 / 继续聊聊”两个动作。
5. 如果生成 AI 解读需要等待，先显示本地短文案，不让用户空等。

今日神谕还有一个重要体验点：当天只能完整翻一次。这对仪式感很好，但必须清楚解释。否则用户会困惑为什么不能重抽。建议在结果页加入一句轻提示：“今天的牌已经为你保留，明天会刷新。”如果 Pro 用户有重置机会，也要明确“重抽会覆盖今天记录吗”。

结果页建议把内容分成 4 块：

1. 今日主题：一句标题。
2. 牌面提醒：这张牌在提醒什么。
3. 今天可做：一个具体行动。
4. 今天先别：一个温和边界。

用户不需要一上来读很长神秘学解释。完整解读可以放二级页面，对话追问可以承接深度需求。

P0：统一今日神谕语言和按钮文案。  
P1：补翻牌选中反馈、震动、失败兜底。  
P1：明确“每日一次”的规则。  
P2：为连续签到增加轻量视觉变化，不做重度任务系统。

## 第 7 页：对话体验与 AI 神谕师

对话模块是产品深度价值的承接点。工程中 `DialogUI` 已经支持聊天列表、背景视频、TTS、快速占卜、三张牌、好友上下文、输入框自动扩展、云端对话状态加载等。这说明对话不是简单文本框，而是核心引擎。

当前体验风险集中在“用户不知道该问什么”。示例图里有 Quick Divination 和推荐问题，这是正确方向。建议把对话首屏分成两类状态：

1. 从每日神谕进入：自动带入今日牌卡片，显示“围绕今天这张牌，你可以问……”。
2. 从底部导航进入：显示“你想问哪一类问题？”并给出情感、事业、自我、今日行动、随机牌阵等入口。

输入框建议保留自由输入，但不要把自由输入作为唯一主动作。占卜类用户常见状态是“有感觉，但说不清”。因此推荐问题、问题模板、情绪标签非常重要。

可以增加的交互：

1. 情绪开场：焦虑、犹豫、想念、困住、期待。
2. 关系开场：关于我、关于 TA、关于我们、关于选择。
3. 追问芯片：再说具体一点、给我行动建议、这张牌为什么、我该等还是行动。
4. 消息长按：复制、保存到记忆、删除、反馈不准。
5. AI 回复流式过程中：显示“正在看牌面 / 正在整理你的问题”，不要只显示转圈。
6. 回复失败：保留用户问题，一键重试，说明是否已保存。

TTS 是亮点，但要避免打扰。默认不自动播放是正确的，建议只在 AI 消息上显示轻量播放按钮，并显示音频生成状态。长消息应分段播放，播放中用户能暂停。

对话还需要安全边界。人格提示词已经写了“不承诺绝对预测、不诊断、不替用户做重大决定”。UI 也应承接这一点：在关系、医疗、法律、投资等敏感问题中，AI 可以温和提醒“这不是专业建议”。不要把安全边界只藏在 prompt 里。

P0：清理测试输入和示例歌词。  
P1：强化对话首屏问题引导。  
P1：补保存、复制、重试、反馈不准。  
P2：增加情绪标签和场景化追问。

## 第 8 页：快速占卜、牌阵与结果页

快速占卜是提升活跃的关键入口。文档中写到：用户 2 小时未对话并从导航栏进入对话时，自动弹出快速占卜入口。这个机制合理，但要谨慎控制打扰。自动弹出如果太强，会像广告；如果只是底部卡片或可展开模块，会更自然。

三张牌占卜结果页结构已经比较清晰：牌面概览、每张牌解释、综合判断、行动建议、推荐追问。建议继续强化“可操作性”。用户做占卜不是为了读一篇文章，而是为了把模糊问题推进一步。每次结果页至少给一个“今天能做的最小行动”。

结果页建议增加以下交互：

1. 折叠/展开：默认展示摘要，详细牌义可展开。
2. 继续追问：每条推荐问题可直接发送到对话。
3. 保存结果：明确保存到历史，并给 toast。
4. 分享卡片：生成一张不泄露隐私的摘要图。
5. 重新提问：允许用户保留牌阵类型，换一个问题重新开始。
6. 标记准确性：准、有点准、不准，用于后续个性化与内容质量评估。

牌阵选择也可以更产品化。当前默认三张牌是“状态、阻碍、建议”，这很通用。可以在 UI 中按问题类型命名：

1. 情感：现在、没说出口的、下一步。
2. 事业：现状、机会/风险、行动。
3. 自我：我在经历什么、我忽略了什么、我可以怎么照顾自己。

这样用户不需要理解塔罗术语，也能感到牌阵贴合问题。

P0：结果页不要出现错别字，如示例中的 `The Word` 应为 `The World`。  
P1：每个结果页补“最小行动”。  
P1：增加追问芯片直发。  
P2：增加准确性反馈与分享卡片。

## 第 9 页：朋友模块与社交关系

朋友模块是 FariApp 的差异化，但也是认知负担最高的模块。它同时包含真实好友、自建好友档案、好友搜索、通讯录邀请、Facebook 邀请、好友请求、双人关系占卜、好友每日牌动态、同步设置、删除与屏蔽。功能很完整，但首屏需要先帮用户理解“朋友模块到底有什么用”。

无好友状态已经有“添加好友”和“创建好友”两个入口，这是正确的。建议把两者解释得更直接：

1. 添加真实好友：双方都在 Fari，可以互相发送邀请和做双人占卜。
2. 创建好友档案：只对你可见，用于记录关系背景和独立占卜。

“创建好友”这个命名容易让用户误以为虚构社交账号。可以考虑改为“创建关系档案”或“添加关系档案”。如果用户要记录恋人、前任、朋友、家人，这个命名会更自然、更隐私。

好友列表设计图里有很大的空白和少量入口，适合早期但信息密度不足。建议列表分为：

1. 顶部：好友请求、双人占卜邀请、搜索/新增。
2. 今日动态：朋友公开同步的每日牌摘要。
3. 真实好友。
4. 我的关系档案。

每个好友卡片建议统一 3 个动作：查看、@ 对话、关系占卜。删除、屏蔽、编辑放到更多菜单里，避免误触。

通讯录和 Facebook 邀请必须“先解释，后授权”。不要点击按钮就调系统权限。推荐弹出底部说明：

“我们只会读取联系人用于帮你发邀请，不会自动添加好友，也不会上传你的通讯录。”

如果做不到“不上传”，必须如实说明读取与存储范围。信任比转化更重要。

P0：明确真实好友和关系档案的差异。  
P0：通讯录权限前置解释。  
P1：好友列表增加分组和空态引导。  
P2：好友动态做成轻量 feed，提高回访。

## 第 10 页：双人关系占卜

双人关系占卜是很有潜力的功能。它把“占卜”从个人内容扩展为社交互动，能自然带来邀请、回访和分享。但这个流程涉及隐私、等待、双方状态同步，UX 必须非常清楚。

当前文档中已经定义了三张牌：

1. 你的内心与看法，仅你可见。
2. 共同揭示，双方可见。
3. 对方的内心与想法，仅对方可见。

这是一个很好的隐私结构。建议在 UI 中把这三张牌做成明显的状态：

1. 仅你可见：带锁图标和“对方看不到”。
2. 双方完成后可见：灰色锁定，显示“等待双方翻牌”。
3. 仅对方可见：只显示背面，不要展示暗示内容。

等待状态要特别打磨。用户发出邀请后，如果只是“等待对方”，很容易流失。建议增加：

1. 邀请剩余时间。
2. 对方是否已打开邀请。
3. 你可以先翻自己的牌。
4. 复制邀请链接/再次提醒。
5. 取消邀请。
6. 等待时可以先和神谕师聊“我为什么想做这次占卜”。

接收方体验也要独立设计。对方收到邀请时，需要立刻明白：这不是暴露隐私，也不是让对方看到你的私密想法。可以写：

“你只会翻开属于你的那张牌。对方看不到你的私牌，只有共同牌会在双方完成后一起显示。”

完成后的结果页建议分为“我的感受、共同关系、下一步互动建议”。避免给出“对方真实想法”的断言，保持 prompt 中的安全原则。

P0：隐私可见性必须在每个状态持续可见。  
P1：等待中增加可做动作。  
P1：接收方入口增加解释页。  
P2：完成后支持双方各自保存私人备注。

## 第 11 页：个人中心、历史与记忆

个人中心目前承载了很多“资产”：资料、历史、通知、反馈、关注、会员、记忆、账号。设计图中的 `My Info` 页面结构清晰，但仍有开发态问题，例如 `No introduction yet`、城市大小写混乱、个人资料项的层级略多。

建议把个人中心分成 4 类：

1. 我的资料：头像、昵称、生日、出生时间、出生地。
2. 我的记录：占卜历史、每日牌、双人占卜、收藏。
3. 我的设置：通知、记忆与隐私、账号安全、语言。
4. 我的权益：Pro、恢复购买、订阅管理。

历史记录是提升留存的关键。用户愿意回来，是因为过去的占卜在今天仍然有意义。当前历史列表已有每日牌、三张牌、双人占卜类型。建议增加：

1. 按时间筛选：今天、本周、本月。
2. 按主题筛选：情感、事业、自我、关系、好友。
3. 收藏/置顶：把重要结果放到顶部。
4. 回访提醒：某条结果可以设置“明天提醒我再看”。
5. 对话延续：从历史结果一键回到对应对话上下文。

AI 记忆是高价值能力，也有高敏感风险。工程中已有记忆管理、隐私开关、清空记忆、候选记忆等。建议把它包装成“你可以控制神谕师记住什么”。在第一次生成记忆时弹出提示：

“我可以记住你的偏好和反复出现的主题，用来让后续解读更贴近你。你可以随时查看、编辑或清空。”

这会把潜在担忧转化为信任资产。

P0：个人中心去掉开发态占位文案。  
P1：历史支持筛选、收藏、继续追问。  
P1：首次记忆生成时做透明提示。  
P2：增加“本周回看”摘要。

## 第 12 页：通知、推送与回访机制

通知是 FariApp 留存的关键，但也是最容易被用户拒绝的权限。APK 声明了 `POST_NOTIFICATIONS`，工程也有 `RemotePushManager`、`AppNotificationScheduler`、通知设置页和远程推送函数。技术准备比较完整，但 UX 上要避免“一上来就要通知权限”。

推荐通知授权时机：

1. 用户完成第一次今日神谕后，提示“明天同一时间提醒你抽今日牌”。
2. 用户发起双人占卜后，提示“对方回应时通知你”。
3. 用户发送 AI 对话后切后台，提示“回复完成时通知你”。
4. 用户保存回访线索后，提示“到时间提醒你回看”。

不要在启动页或登录页直接请求系统通知权限。用户还不知道通知有什么价值，授权率会低。

通知设置页应该让用户理解每类通知的频率和内容。建议分为：

1. 每日神谕提醒：每天一次，可选时间。
2. 对话回复：只在 AI 回复完成或离线队列完成时提醒。
3. 占卜回访：用户主动保存后提醒。
4. 好友互动：好友请求、双人占卜邀请、共同结果完成。
5. 系统与活动：默认关闭或低频。

文案上要避免“开启/关闭”之外没有解释。每个开关下加一行灰色说明，能显著降低误解。

当前剩余事项中提到离线聊天兜底定时函数 `processStaleDialogueReplyJobs` 还未线上部署闭环。这会影响“切后台后回复通知”的体验。发布前必须完成真机链路：发送消息、切后台、服务端完成、收到推送、点击回到正确对话。

P0：通知权限改为场景触发。  
P0：完成推送点击落地页真机验收。  
P1：通知设置加频率说明。  
P2：允许用户选择每日提醒时间和安静时段。

## 第 13 页：权限、隐私与信任

FariApp 涉及多种敏感数据：生日、出生时间、出生地、好友关系、通讯录、Facebook 好友、AI 对话、记忆、占卜历史、推送 token、IAP 订阅。用户体验上，信任不是“隐私政策页面”能单独解决的，而要在每个请求发生前解释清楚。

APK 当前声明了 `READ_CONTACTS`、`POST_NOTIFICATIONS`、`AD_ID`、`ACCESS_ADSERVICES_*`、`BILLING`、`READ_EXTERNAL_STORAGE`、`WRITE_EXTERNAL_STORAGE` 等权限。建议发布前逐项确认是否必要。尤其是广告 ID 和 AdServices 权限，如果当前没有广告归因或广告投放功能，会增加商店审核和用户信任成本。外部存储权限在高版本 Android 上也需要重新评估，头像选择通常可以使用系统 Photo Picker 或 scoped storage。

需要加的隐私交互：

1. 出生信息说明：用于更贴合的占星/关系解读，不会公开给好友。
2. 通讯录说明：只用于邀请，不会自动添加或公开。
3. 每日牌同步说明：只同步摘要，不同步完整解读。
4. AI 记忆说明：用户可查看、编辑、关闭、清空。
5. 双人占卜说明：私牌只对本人可见，共同牌双方完成后可见。
6. 删除账号说明：会删除哪些数据、保留哪些法务/交易记录。

建议在设置中增加“隐私中心”一级入口，包含：

1. 我的公开资料。
2. 每日牌同步范围。
3. AI 记忆设置。
4. 好友与屏蔽。
5. 数据导出/删除。
6. 权限管理引导。

这样隐私不再是被动防御，而是产品可信度的一部分。

P0：清理不必要权限。  
P0：所有敏感权限增加前置解释。  
P1：新增隐私中心聚合入口。  
P2：增加“查看我对好友公开了什么”的预览。

## 第 14 页：本地化、文案与品牌一致性

文案是当前最需要发布前打磨的部分。工程中存在 I2 本地化资源，但实际 prefab、示例图和部分运行时文案仍混杂中文、英文、葡语，以及测试内容。对一个依赖文字情绪的产品，这不是小问题。

需要立即处理的文案问题：

1. APK 应用名是 `fariapp`，建议改为 `Fari` 或正式品牌名。
2. 包名是 `com.canchentechnology.fariapp`，工程文档中又出现 `com.canchentechnology.fari`，需要确认发布包一致性。
3. 示例中有 `Never gonna give you up...` 这类测试歌词，必须移除。
4. 部分按钮仍显示 `Button`。
5. `The Word` 应修正为 `The World`。
6. 英文界面中的 `Mine` 建议改为 `Me`。
7. 中文、英文、葡语不要在同一个版本混用。

文案风格建议保持“克制、清醒、温柔、神秘但不装腔”。工程中的 Nocturne Oracle 人格设定很好，UI 文案也应同一口径。例如：

错误方向：

“你一定会得到答案”

推荐方向：

“先抽一张牌，看见此刻的线索”

错误方向：

“允许通讯录权限”

推荐方向：

“找到也在 Fari 的朋友。我们不会自动添加联系人。”

错误方向：

“升级 Pro 解锁所有功能”

推荐方向：

“保留更多对话与牌阵，让神谕师持续理解你”

建议建立一份 `UX Copy Guideline`，规定：

1. 主语言。
2. 称呼方式。
3. 禁止承诺预测。
4. 错误提示模板。
5. 权限解释模板。
6. 按钮文案长度。
7. 中英文术语表。

P0：清理所有测试文案和占位文案。  
P0：统一正式品牌名。  
P1：建立文案风格表。  
P2：为多语言准备完整 key，而不是 prefab 硬编码混用。

## 第 15 页：视觉设计与可读性

视觉气质是 FariApp 的优势。暗色基底、橙金高亮、卡牌与占卜师角色能形成记忆点，适合“夜间、私密、情绪整理”的使用场景。用户打开首页时，很容易知道这不是普通聊天工具。

但视觉上也有几个风险：

1. 橙金色承担太多角色：标题、按钮、边框、选中态、强调文字都用它，层级可能变平。
2. 深色卡片和深色背景对比不足，部分页面信息密度低时显得空。
3. 首页人物和牌桌很强，进入朋友/设置等工具页后视觉骤降，体验割裂。
4. 示例图中的底部导航有时是深灰卡片，有时有橙色描边，风格需统一。
5. 部分长文字在卡片中截断，未说明能否展开。

建议建立视觉层级：

1. 橙金：主动作、当前选中、牌名。
2. 白色：主要正文。
3. 灰紫：辅助说明。
4. 红色：危险动作，如删除、清空。
5. 绿色：在线或成功状态，谨慎使用。

卡片圆角和间距也要统一。当前很多页面使用大卡片，适合沉浸，但设置页和列表页不宜全都厚重。朋友列表、历史列表、通知设置可以更紧凑，减少空屏感。

图片资产方面，塔罗牌素材已经比较完整。建议做一套卡牌加载占位，不要在图片加载失败时只显示空框或默认图。对占卜产品来说，牌图缺失会严重破坏仪式感。

P0：统一底部导航视觉和主按钮样式。  
P1：建立颜色用途规范。  
P1：长文本增加展开/收起。  
P2：为不同神谕师角色做微差异背景，而不是完全换风格。

## 第 16 页：状态反馈、错误处理与离线体验

工程中已经做了不少离线兜底：好友、每日牌、历史、记忆、通知设置、自建好友等都在文档中记录了本地缓存和稍后同步。这是很好的技术基础。UX 需要把这些能力表达出来。

用户不关心“Firestore 初始化失败”，用户关心“我刚刚做的事有没有丢”。因此所有云端相关行为都应有三类反馈：

1. 成功：已保存。
2. 本地成功云端待同步：已保存到本地，网络恢复后同步。
3. 失败且未保存：未保存，请重试。

当前 toast 已有类似提示，但建议统一样式和语气。不要不同模块各写一套“失败/稍后同步”。

需要补齐的状态：

1. 每日神谕生成中：显示预计步骤，不只转圈。
2. AI 回复中：显示流式输入、取消发送、后台通知说明。
3. 好友搜索中：空结果、网络失败、已发送、已添加、已屏蔽。
4. 双人占卜中：邀请已发、对方已看、对方已翻、共同结果可揭示。
5. IAP 购买中：连接商店、等待验证、凭证提交失败、待配置。
6. 头像上传中：压缩、上传、保存资料、失败重试。

离线模式建议做成显性能力，而不是故障兜底。例如：

“当前网络不稳定，你仍可查看本地历史和继续写下问题。联网后会同步。”

这比“网络错误”更有陪伴感。

P0：统一所有保存/同步/失败 toast。  
P1：关键流程加骨架屏和重试。  
P1：离线可用范围显性化。  
P2：增加后台任务中心，显示待同步项目。

## 第 17 页：会员、付费与价值表达

Pro 功能已经接入 Unity IAP、后端 receipt 校验、会员状态缓存和额度控制。当前 Pro 权益包括每日牌、对话消息、牌阵占卜、神谕师角色、记忆与历史等。付费基础具备，但 UX 上需要把“为什么值得付费”讲得更具体。

占卜类产品的付费点不应只说“更多额度”。更好的表达是“持续陪伴”和“更完整的个人洞察”。建议把 Pro 页面权益改为：

1. 更多深度对话：每天更充分地追问和整理问题。
2. 更多牌阵：不只每日牌，可做情感、事业、自我探索。
3. 多位神谕师视角：塔罗、占星、冥想，不同问题有不同方式。
4. 记忆与回访：让神谕师记住你的主题、关系和行动线索。
5. 完整历史：回看过去的牌，看到自己的变化。

付费触发时机要自然。不要在用户还没感受到价值时弹 Pro。推荐触发：

1. 用户连续追问多轮后。
2. 用户想做第二个牌阵时。
3. 用户查看较早历史或高级回访时。
4. 用户想切换神谕师角色时。
5. 用户达到免费额度，但当前草稿必须保留。

工程中已经处理了“会员/额度拦截保留输入草稿”，这是非常重要的体验点。建议继续做两件事：

1. 额度用完前提前提示，而不是发送时突然拦截。
2. Pro 弹窗显示“你今天已使用 5/5 次”，让限制透明。

IAP 价格目前 public config 中 `priceLabel` 为空，发布前需要从商店返回真实价格或在配置中填写展示价格。否则购买按钮容易显得未完成。

P0：Pro 页面显示真实价格和恢复购买入口。  
P1：付费触发要保留上下文和草稿。  
P1：从“更多额度”升级为“更完整陪伴”。  
P2：增加试用或首周折扣策略。

## 第 18 页：性能、包体与发布体验

APK 约 162MB，对一个移动端占卜应用来说偏大。Unity、IL2CPP、Firebase、Facebook、IAP、视频背景、塔罗图片、YooAsset 都会推高体积。包体大不是绝对问题，但会增加下载门槛，尤其在移动网络和新用户冷启动阶段。

建议做包体拆分：

1. 首包只保留启动、登录、今日神谕、基础对话、核心牌图。
2. 好友、Facebook、完整塔罗高清图、部分视频素材、扩展角色素材可按需下载。
3. YooAsset 已存在，可用它承接资源分包和增量更新。
4. 检查是否同时包含 arm64-v8a 和 armeabi-v7a。若目标市场允许，可考虑只发 arm64 以减小包体。
5. 检查是否有未使用 Firebase/Google/Facebook 依赖引入权限和体积。

启动性能上，视频背景和大图沉浸感很好，但要控制首屏可交互时间。建议指标：

1. 冷启动 3 秒内看到品牌/启动视觉。
2. 6 秒内可进入今日神谕或登录。
3. 热更新检查超过 3 秒显示明确状态。
4. 视频首帧未准备好时显示静态背景，不黑屏。

发布体验上，当前应用名 `fariapp`、版本 `0.1`、包名不一致风险、权限较多、真机验证未完成，说明还不适合直接面向用户发布。至少需要完成：

1. Android 真机登录、抽牌、AI 对话、好友请求、双人占卜、通知权限、推送点击。
2. 弱网和断网测试。
3. IAP 沙盒购买与恢复。
4. 删除账号与隐私设置。
5. 多尺寸屏幕适配。

P0：发布前完成真机路径验收。  
P0：应用名、图标、包名、权限清单定版。  
P1：首包体积优化。  
P2：建立启动性能埋点。
